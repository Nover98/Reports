
**moduleResolution** — определяет, как TypeScript ищет модуль по пути в `import`/`export`.

То есть эта настройка влияет не только на то, найдётся ли файл без расширения, но и вообще на правила резолва:

- можно ли писать относительные импорты без `.js`
- как искать файл: по правилам Node.js или по правилам bundler
- нужно ли учитывать `package.json`, в том числе поля `exports` и `imports`
- как резолвить пакеты из `node_modules`
- будет ли редактор подчёркивать импорт как ошибочный

Надо понимать, что в настоящем ESM-мире, будь то браузер или Node.js, путь обычно должен быть указан более буквально. Браузер ожидает корректный URL, а Node.js в ESM-режиме обычно тоже хочет явный путь к файлу.

Поэтому когда мы пишем импорт без расширения, импортируем папку в расчёте на `index.js` или вообще подключаем что-то нестандартное вроде картинки в код, это часто работает не потому, что так устроен «чистый» ESM, а потому что у нас есть сборщик:

- Vite
- Webpack
- esbuild
- Rollup

Сборщик достаточно умный, чтобы:

- посмотреть в `package.json`
- учесть `exports`
- попробовать разные расширения: `.js`, `.ts`, `.tsx` и другие
- догадаться, что нужно взять `index.js` или другой подходящий файл

То есть в «чистом» ESM-мире путь обычно должен быть более точным, а многие привычные сокращения — это уже удобство, которое даёт bundler.

## Почему TypeScript начинает требовать расширение

Если поставить, например, `moduleResolution: "nodenext"` или работать в Node-ориентированном ESM-сценарии, TypeScript старается вести себя ближе к Node.js.

А Node.js в таком режиме обычно требует явный относительный путь. Поэтому импорт вроде:

```ts
import { sum } from './math';
```

может стать ошибкой, и TypeScript попросит написать так:

```ts
import { sum } from './math.js';
```

Важно: в таком режиме TypeScript обычно просит писать именно `.js`, а не `.ts`.

То есть если на диске лежит `math.ts`, импорт в коде всё равно часто должен выглядеть так:

```ts
import { sum } from './math.js';
```

Это не ошибка логики. TypeScript понимает, что в runtime Node.js будет искать `math.js`, но при анализе исходников умеет сопоставить этот путь с `math.ts` и взять типы именно оттуда.

То есть для выполнения путь пишется как будущий JS-файл, а для проверки типов TypeScript всё равно находит исходный TS-файл.

## Коротко про разницу режимов

- `node10` / старые node-режимы — более старые правила поиска модулей
- `nodenext` — поведение ближе к современному Node.js с ESM/CommonJS-правилами
- `bundler` — режим для проектов, где реальные импорты потом обрабатывает сборщик, поэтому TypeScript не навязывает слишком жёсткие правила Node.js
