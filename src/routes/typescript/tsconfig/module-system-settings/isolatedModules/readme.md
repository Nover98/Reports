`isolatedModules` — это флаг для сценариев, где TypeScript-код транспилируется по одному файлу за раз, без полноценного анализа всей программы.

Так работают некоторые инструменты вроде Babel, esbuild, swc и другие быстрые транспайлеры. Они не запускают полноценную проверку типов и не строят полную картину всех файлов проекта. Вместо этого они берут файл, убирают из него TypeScript-специфику и превращают его в JavaScript.

Чтобы это было безопасно, каждый файл должен быть таким, чтобы его можно было корректно обработать изолированно, не заглядывая в другие файлы.

Проблема в том, что в TypeScript есть конструкции, для которых одного файла недостаточно. Без анализа других файлов транспайлер может просто не понять, что именно перед ним, и сгенерировать неправильный код.

Например:

```ts
// types.ts
export type User = { id: number };

// index.ts
export { User } from './types';
```

Если инструмент обрабатывает только `index.ts`, он не знает, `User` — это тип или реальное значение. А значит он не может надёжно понять, нужно ли оставлять этот export в итоговом JS.

Поэтому с `isolatedModules` TypeScript заставляет писать такие вещи явно и избегать конструкций, которые нельзя безопасно транспилировать пофайлово.

Ещё один классический пример — некоторые сценарии с `const enum`. Там тоже может требоваться знание других файлов или всей программы целиком.

То есть смысл флага не в том, что он что-то оптимизирует сам по себе, а в том, что он заранее проверяет: не используешь ли ты такой TypeScript-код, который сломает single-file transpilation.

Разница между `isolatedModules` и `verbatimModuleSyntax` такая:

- `verbatimModuleSyntax` — про то, чтобы TypeScript не пытался сам догадываться, какие импорты и экспорты относятся только к типам, а какие должны остаться в runtime. С ним это нужно писать явно через `import type` и `export type`.
- `isolatedModules` — про то, можно ли вообще безопасно транспилировать файл изолированно, без знания всей программы.

То есть между ними есть пересечение, но это не один и тот же флаг и один не заменяет другой.

`isolatedModules` может поймать некоторые проблемные конструкции вокруг импортов и экспортов, но он не означает: «веди себя точно так же, как `verbatimModuleSyntax`».

Поэтому вывод такой:

- `isolatedModules` не включает в себя `verbatimModuleSyntax`
- `verbatimModuleSyntax` не включает в себя `isolatedModules`
- в современных проектах их часто имеет смысл включать вместе

## Где поведение может совпасть

Например, в таком случае:

```ts
// types.ts
export type User = { id: number };

// index.ts
export { User } from './types'; // это не импорт, а реэкспорт!!!)
```

И `isolatedModules`, и `verbatimModuleSyntax` по сути подведут к одному и тому же исправлению:

```ts
export type { User } from './types';
```

Но причина у них разная:

- `verbatimModuleSyntax` требует писать type-only export явно
- `isolatedModules` проверяет, можно ли безопасно транспилировать файл, не зная содержимое `types.ts`

## Где поведение уже может отличаться

Например:

```ts
import { User } from './types';

const user: User = { id: 1 };
```

Здесь `verbatimModuleSyntax` обычно захочет, чтобы импорт был записан явно как type-only:

```ts
import type { User } from './types';
```

А вот `isolatedModules` в таком случае может не ругаться, потому что внутри этого же файла TypeScript видит, что `User` используется только как тип, и для пофайловой трансформации этого уже достаточно.
