

**moduleDetection** — определяет, как TypeScript решает, считать файл модулем или глобальным скриптом.

Это важно, потому что у скриптов и модулей разное поведение.

## Скрипт и модуль — в чём разница

- **Скрипт** (`script`) живёт в общем глобальном пространстве имён. Если в двух script-файлах объявить одинаковую сущность, можно получить ошибки вроде `Duplicate identifier`.
- **Модуль** (`module`) изолирован. Всё внутри файла остаётся внутри файла, пока это явно не экспортировано.

Проще говоря: у модуля есть границы, у скрипта — нет.

## Какие есть режимы

### `auto` — поведение по умолчанию

Это текущий режим по умолчанию.

TypeScript определяет модуль не только по `import`/`export`, но и по контексту проекта. Например, он может считать файл модулем, если:

- в файле есть `import` или `export`
- используется `import.meta`
- это JSX-файл в современном JSX-режиме
- проект работает в современном ESM-контексте, например через `module: "node16"` / `"nodenext"` и `package.json` с `"type": "module"`

То есть `auto` пытается подстроиться под современный JavaScript-мир.

### `legacy`

Это старое классическое поведение TypeScript.

Логика простая:

- есть `import` или `export` — значит модуль
- нет `import` и `export` — значит глобальный скрипт

Если файл нужно сделать модулем вручную (например side effects module, который ничего не экспортирует/импортирует и подключается как `import "./create-menu.js"`), но в нём нет реальных импортов и экспортов, обычно добавляют пустой:

```ts
export {};
```

### `force`

Самый жёсткий и предсказуемый режим.

С ним каждый файл, кроме .d.ts, считается модулем, даже если внутри вообще нет `import` и `export`.

Это удобно, если нужно полностью убрать случайные пересечения имён между файлами и жить только в модульной модели.

## Когда этот флаг реально нужен

`moduleDetection` становится важным, когда в проекте есть файлы без `import`/`export`, и нужно понять, как TypeScript будет их трактовать:

- как глобальные script-файлы
- или как нормальные изолированные модули

От этого зависит:

- будет ли файл делить глобальную область видимости с другими файлами
- появятся ли ошибки из-за дублирования имён
- можно ли использовать конструкции, которым нужен именно модульный контекст









## История: почему вообще понадобился этот флаг

### 1. Изначально JavaScript жил в мире скриптов

Когда TypeScript появился в 2012 году, стандартных ESM-модулей в JavaScript ещё не было. В браузере код обычно подключался так:

```html
<script src="jquery.js"></script>
<script src="utils.js"></script>
<script src="app.js"></script>
```

Такие файлы выполнялись как обычные скрипты. Всё, что объявлялось в одном файле, могло влиять на другие. Это и был обычный глобальный сценарий работы JavaScript того времени.

### 2. TypeScript рано добавил модульный синтаксис

Разделять код по файлам всё равно было нужно, поэтому TypeScript довольно рано поддержал `import`/`export` и компилировал такой код в старые модульные системы:

- `CommonJS` — для Node.js
- `AMD` — для браузерных загрузчиков вроде RequireJS

То есть модульная модель в TypeScript появилась раньше, чем нативные ESM-модули стали реальностью в самом JavaScript.

### 3. Потом появился ESM, но старый код никуда не делся

В 2015 году в JavaScript появился стандартный модульный формат ESM.

Но к этому моменту уже существовало огромное количество кода:

- старого TypeScript-кода, завязанного на предыдущие правила
- обычных JS/TS script-файлов без модулей

Если бы TypeScript резко объявил: «с этого дня все файлы по умолчанию считаются модулями», это сломало бы совместимость с большим количеством старых проектов.

Причина простая:

- старые проекты рассчитывали на глобальный scope
- новый модульный подход изолировал бы файлы
- из-за этого сразу вылезли бы ошибки и изменилось бы поведение существующего кода

### 4. Поэтому TypeScript долго держался за старую логику

По этой причине TypeScript много лет придерживался безопасного правила:

> если в файле нет признаков модуля, значит это script

Именно это поведение сегодня называется `legacy`.

Оно существовало очень долго не потому, что это идеальная модель, а потому, что она не ломала старый код.

### 5. Когда экосистема стала современной, появился `moduleDetection`

Со временем мир изменился:

- появились нативные ESM-модули
- Node.js получил поддержку ESM
- bundler’ы и современные фронтенд-проекты стали жить почти полностью в модульной модели

В такой среде правило «нет `import`/`export` — значит script» стало слишком грубым. Файл мог жить в ESM-проекте и при этом не содержать ни одного импорта, но по смыслу всё равно быть модулем.

Именно как ответ на это в TypeScript 4.7 появился `moduleDetection`.

Он добавил более современную логику:

- `legacy` — старое поведение ради совместимости
- `auto` — учитывать не только содержимое файла, но и контекст проекта
- `force` — считать все файлы модулями
