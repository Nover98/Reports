# `types`

`types` — это ограничитель для глобальных типов из `node_modules/@types`.

Если `lib` описывает стандартные возможности языка и среды, например `Math` или `fetch`, то `types` управляет тем, какие сторонние пакеты с типами будут автоматически видны во всём проекте.

## Что происходит по умолчанию

Если не трогать эту настройку, TypeScript ищет все **видимые** пакеты в `node_modules/@types` и автоматически подключает их глобальные объявления.

Например:

- установлены типы для `node` — глобалы Node.js будут доступны везде;
- установлены типы для `jest` или `mocha` — функции вроде `test()` или `describe()` будут видны даже в файлах, где тесты вообще не пишутся.

## Что делает `types`

Если добавить поле `types` в `tsconfig.json`, TypeScript перестаёт автоматически подтягивать все пакеты из `@types` и подключает только те, которые явно перечислены в массиве.

Например:

- если пакет называется `@types/node`, в массиве нужно написать `"node"`;
- если пакет называется `@types/lodash`, в массиве нужно написать `"lodash"`.

Префикс `@types/` при этом отбрасывается.

Но здесь есть важный нюанс: записи в `types` не обязаны ссылаться только на пакеты из `@types`.

Например, запись `"vite/client"` работает не потому, что существует пакет `@types/vite`, а потому, что TypeScript умеет резолвить такую запись как путь к типам внутри самого пакета `vite`.

То есть:

- `"node"` обычно резолвится в `@types/node`;
- `"jest"` — в `@types/jest`;
- `"vite/client"` — в типы, которые сам пакет `vite` уже поставляет внутри себя.

Поэтому `types` лучше понимать не как «список пакетов только из `@types`», а как список type entrypoints, которые TypeScript должен явно подключить.

## Что `types` не ограничивает

Важно понимать: `types` ограничивает только **автоматическое подключение глобальных пакетов** из `node_modules/@types`.

Эта настройка **не запрещает** TypeScript получать типы из других источников.

Например, типы всё равно могут попасть в проект через:

- локальные `.d.ts` файлы, которые входят в проект через `files`, `include` или просто лежат в области видимости текущего `tsconfig.json`;
- пакеты, которые экспортируют типы сами, без `@types`, через свои `.d.ts` файлы или поле `types` / `typings` в `package.json`;
- обычные `import`/`export`, когда ты явно импортируешь модуль, а TypeScript вместе с ним подтягивает и его типы;
- `/// <reference types="..." />` и другие явные ссылки на объявления типов.

Например, даже если в `types` не указан `node`, типы Node.js всё равно могут появиться в проекте, если:

- в проект попал локальный `.d.ts`, где объявлены node-специфичные типы;
- какой-то пакет явно ссылается на `node`-типы;
- ты импортировал пакет, который сам тянет эти типы как зависимость своих объявлений.

То есть `types` — это не «полный запрет на типы», а только фильтр для **автоматически подключаемых** `@types`-пакетов.

## Зачем ограничивать этот список

- **Защита от «грязного» автодополнения.** Если в проекте есть типы и для Node.js, и для браузера, редактор может предлагать `process.exit()` там, где этого API на самом деле не существует.
- **Избежание конфликтов имён.** Две разные библиотеки могут объявить одну и ту же глобальную переменную. Если TypeScript подключит обе, он начнёт ругаться на конфликт, хотя фактически используется только одна из них.
- **Скорость.** Чем меньше типов TypeScript нужно обрабатывать, тем быстрее работает проверка.

## Почему браузерные типы идут через `lib`, а типы Node.js — нет

Это не «несправедливость», а разница в источнике этих типов.

Браузерные API входят в набор веб-стандартов, поэтому TypeScript поставляет их описания вместе со встроенными `lib`-файлами.

Node.js — это отдельная платформа со своим жизненным циклом и своим набором API. Поэтому её типы поставляются отдельно, обычно через пакет `@types/node`.

Исторически это тоже логично: TypeScript изначально создавался прежде всего как инструмент для браузерной разработки, поэтому браузерное окружение с самого начала считалось основным сценарием.
