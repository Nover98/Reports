**lib** — это список встроенных наборов типов, которые TypeScript подключает для проекта.

Принимает массив строк, где каждая строка — это название набора типов. Эта настройка принимает только набор готовых значений, указанных на сайте TypeScript. Свои значения туда добавлять нельзя.

Проще говоря, эта настройка говорит компилятору:

- какие глобальные объекты и функции считать существующими
- какие возможности JavaScript доступны в текущей среде
- есть ли в проекте браузерные API вроде `window`, `document`, `fetch`
- есть ли современные JS-структуры вроде `Promise`, `Map`, `Set`

## Зачем нужна `lib`

TypeScript проверяет не только пользовательский код, но и знает типы встроенных API.

Например, чтобы понимать такие вещи:

- что `document.querySelector()` существует
- что `Promise` существует
- что у массива есть `flat()`
- что у строки есть `replaceAll()`

Все эти знания берутся не «из воздуха», а из подключённых `lib`.

Если `lib` не указана явно, TypeScript сам выбирает набор библиотек по умолчанию, в первую очередь исходя из настройки `target`.

Например:

- более старый `target` → меньше современных JS API
- более новый `target` → больше современных JS API

Если `lib` указана явно, TypeScript полностью отключает все автоматические наборы типов и будет использовать только те библиотеки, которые явно перечислены в массиве. То есть если массив пустой, `TS Server` сразу начнёт подчёркивать почти всё.

## Значения

### `DOM`

Набор типов для браузерного окружения: `window`, `document`, `HTMLElement` и т.д.

### `ES****`

Если в коде используются `Promise`, `Map`, `Set`, `Symbol` и т.д., то нужен соответствующий ES-набор, например `ES2023`.

### `DOM.Iterable`

Нужен, чтобы TypeScript понимал, что некоторые DOM-объекты можно использовать как итерируемые коллекции.

Это дополнительный набор типов для DOM-объектов, которые можно перебирать как iterable-коллекции.

Он добавляет TypeScript-знание о том, что некоторые браузерные штуки поддерживают:

- `for...of`
- `entries()`
- `keys()`
- `values()`
- `Symbol.iterator`

## Почему это вынесено отдельно

Причина есть, и она историческая и архитектурная.

`DOM` и `DOM.Iterable` разделены потому, что это два разных слоя описания:

- `DOM` — говорит, какие браузерные объекты вообще существуют
- `DOM.Iterable` — говорит, какие из них поддерживают iterable-протокол
